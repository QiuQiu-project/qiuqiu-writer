# 增量更新修复说明

## 问题描述

之前的实现中，多用户编辑文档时，同步机制会发送**完整内容**而不是**增量操作**，导致：
- 用户B看到用户A的所有编辑内容被"粘贴"过来
- 无法实现真正的增量更新
- 用户体验不佳

## 解决方案

### 1. 后端：计算差异并生成操作

**文件**: `backend/src/memos/api/services/sharedb_service.py`

#### 新增方法：`_calculate_diff_operations()`

使用 `difflib` 计算两个内容之间的差异，生成操作列表：

```python
def _calculate_diff_operations(self, old_content: str, new_content: str) -> List[Dict[str, Any]]:
    """
    计算两个内容之间的差异，生成操作列表
    生成的操作类型：
    - insert_text: 插入文本
    - delete_text: 删除文本
    - replace_text: 替换文本（删除+插入）
    """
```

#### 修改 `sync_document()` 方法

在同步文档时：
1. 计算差异并生成操作
2. 如果操作数 <= 100，发送增量操作
3. 如果操作数 > 100，回退到完整内容更新（避免性能问题）

```python
# 计算差异并生成操作（增量更新）
operations = []
if document and "content" in document:
    previous_content = document.get("previous_content") or server_content
    new_content = document["content"]
    
    if previous_content != new_content:
        operations = self._calculate_diff_operations(previous_content, new_content)
    
    document["previous_content"] = new_content

# 如果差异太大，回退到完整内容更新
if len(operations) > 100:
    operations = []

# 广播更新
if operations:
    # 发送增量操作
    await self._broadcast_update(document_id, {
        "type": "document_synced",
        "operations": operations,  # 发送操作
        "full_content": False
    })
else:
    # 发送完整内容
    await self._broadcast_update(document_id, {
        "type": "document_synced",
        "content": document["content"],
        "operations": [],
        "full_content": True
    })
```

### 2. 前端：应用操作而不是替换内容

**文件**: `frontend/src/utils/sharedbClient.ts`

#### 新增方法：`applyOperations()`

应用操作到当前内容：

```typescript
private applyOperations(content: string, operations: any[]): string {
  let result = content;
  
  // 按位置从后往前排序，避免位置偏移问题
  const sortedOps = [...operations].sort((a, b) => b.position - a.position);
  
  for (const op of sortedOps) {
    switch (op.type) {
      case 'insert_text':
        result = result.slice(0, pos) + op.text + result.slice(pos);
        break;
      case 'delete_text':
        result = result.slice(0, pos) + result.slice(pos + op.length);
        break;
      case 'replace_text':
        result = result.slice(0, pos) + op.text + result.slice(pos + op.length);
        break;
    }
  }
  
  return result;
}
```

#### 修改 `handleMessage()` 方法

处理文档更新消息时：
1. 检查是否有操作（operations）
2. 如果有操作且 `full_content` 为 false，应用操作
3. 否则，使用完整内容更新

```typescript
case 'document_synced':
  const operations = message.operations || [];
  const fullContent = message.full_content !== false;
  
  if (operations.length > 0 && !fullContent) {
    // 应用增量操作
    currentContent = this.applyOperations(currentContent, operations);
    callback(currentContent, version);
  } else if (content !== undefined) {
    // 使用完整内容更新
    callback(content, version);
  }
  break;
```

## 工作流程

### 场景：用户A和用户B同时编辑

1. **用户A编辑**：
   - 输入 "Hello" → "Hello World"
   - 调用 `/documents/sync` API
   - 服务器计算差异：`insert_text(position=5, text=" World")`
   - 服务器广播操作给所有订阅的客户端

2. **用户B收到更新**：
   - WebSocket 收到 `document_synced` 消息
   - 检查是否有 `operations`
   - 如果有，调用 `applyOperations()` 应用操作
   - 用户B看到的是增量更新，而不是完整内容替换

### 操作类型

#### insert_text
```json
{
  "type": "insert_text",
  "position": 5,
  "text": " World"
}
```

#### delete_text
```json
{
  "type": "delete_text",
  "position": 5,
  "length": 6
}
```

#### replace_text
```json
{
  "type": "replace_text",
  "position": 5,
  "length": 5,
  "text": "World"
}
```

## 优势

1. **真正的增量更新**：只发送变化的部分，不是完整内容
2. **更好的用户体验**：用户看到的是逐步更新，而不是突然替换
3. **减少网络传输**：操作通常比完整内容小得多
4. **保持编辑状态**：应用操作不会影响用户正在编辑的内容

## 回退机制

如果差异太大（操作数 > 100），系统会：
1. 回退到完整内容更新
2. 标记 `full_content: true`
3. 前端使用完整内容更新

这确保了：
- 小变化使用增量更新（快速、精确）
- 大变化使用完整内容更新（可靠、简单）

## 测试方法

1. 打开两个浏览器窗口，访问同一章节
2. 在窗口A中输入少量内容（例如：添加一个词）
3. 观察窗口B：
   - 应该看到增量更新（只添加了那个词）
   - 不应该看到完整内容被替换
   - 控制台应该显示：`📝 [ShareDB] 应用增量操作: X 个操作`

## 注意事项

1. **操作顺序**：操作按位置从后往前应用，避免位置偏移
2. **边界检查**：确保位置和长度在有效范围内
3. **错误处理**：如果应用操作失败，回退到完整内容更新
4. **性能考虑**：操作数过多时自动回退
