# 基于文件名的逐章分析功能设计

## 功能概述

实现一个智能的章节分析流程：
1. **首次分析**：使用文件名称创建新作品
2. **逐章分析**：每次分析一章时，AI输出该章的大纲和细纲
3. **自动插入**：将分析结果（包括大纲、细纲）插入到作品的章节中
4. **卷支持**：如果章节属于某个卷，自动创建卷
5. **续分析**：第二次分析时，根据文件名自动找到对应的作品并继续插入

## 工作流程

### 第一次分析（创建作品）

```
用户上传文件 "我的小说.txt"
  ↓
调用 API: POST /ai/analyze-chapter-by-file
  ↓
参数: {
  "file_name": "我的小说.txt",
  "content": "第一章内容...",
  "chapter_number": 1,
  "volume_number": 1 (可选)
}
  ↓
后端处理:
  1. 根据 file_name 查找作品（work_metadata.source_file）
  2. 如果不存在，创建新作品（标题=文件名）
  3. 分析章节内容（使用 chapter_analysis prompt）
  4. 解析AI返回的JSON（包含大纲和细纲）
  5. 创建章节并保存大纲和细纲到 chapter_metadata
  6. 如果有卷号，确保卷存在
  ↓
返回: {
  "work_id": 123,
  "work_title": "我的小说",
  "chapter_id": 456,
  "chapter_number": 1,
  "outline": {...},
  "detailed_outline": {...}
}
```

### 后续分析（继续插入）

```
用户继续分析同一文件的下一章
  ↓
调用相同的 API，但 chapter_number 不同
  ↓
后端处理:
  1. 根据 file_name 找到已存在的作品
  2. 检查章节是否已存在（避免重复）
  3. 分析新章节
  4. 插入到同一作品中
  ↓
返回: 新章节信息
```

## API 接口设计

### POST /ai/analyze-chapter-by-file

**功能**：根据文件名分析单章并插入到作品

**请求参数**：
```json
{
  "file_name": "我的小说.txt",  // 文件名，用于查找或创建作品
  "content": "第一章的完整内容...",  // 章节内容
  "chapter_number": 1,  // 章节号
  "volume_number": 1,  // 卷号（可选，默认为1）
  "settings": {  // AI设置（可选）
    "model": "gpt-4",
    "temperature": 0.7,
    "max_tokens": 4000
  }
}
```

**响应格式**（流式响应）：
```
data: {"type": "start", "message": "开始分析章节"}
data: {"type": "chunk", "content": "..."}
data: {"type": "work_found", "work_id": 123, "work_title": "我的小说"}
data: {"type": "work_created", "work_id": 123, "work_title": "我的小说"}
data: {"type": "chapter_analyzed", "chapter_number": 1, "title": "第一章"}
data: {"type": "chapter_inserted", "chapter_id": 456, "chapter_number": 1}
data: {"type": "done", "message": "分析完成"}
```

**响应数据**（最终）：
```json
{
  "work_id": 123,
  "work_title": "我的小说",
  "chapter_id": 456,
  "chapter_number": 1,
  "title": "第一章 标题",
  "outline": {
    "core_function": "...",
    "key_points": [...],
    "visual_scenes": [...],
    "atmosphere": [...],
    "hook": "..."
  },
  "detailed_outline": {
    "sections": [...]
  }
}
```

## 实现要点

### 1. 文件名存储和查找

在 `Work.work_metadata` 中存储源文件信息：
```python
work_metadata = {
    "source_file": "我的小说.txt",
    "source_file_hash": "...",  # 可选：文件哈希，用于更精确匹配
    "analysis_mode": "file_based"  # 标记这是基于文件的分析
}
```

查找逻辑：
```python
async def find_work_by_filename(self, file_name: str, user_id: int) -> Optional[Work]:
    """根据文件名查找作品"""
    stmt = select(Work).where(
        and_(
            Work.owner_id == user_id,
            Work.work_metadata['source_file'].astext == file_name
        )
    )
    result = await self.db.execute(stmt)
    return result.scalar_one_or_none()
```

### 2. Prompt 模板

使用现有的 `chapter_analysis` prompt 模板（已在 `init_chapter_analysis_prompt.py` 中定义），该模板要求AI输出：
- 章节基本信息
- 大纲（outline）：包含 core_function、key_points、visual_scenes、atmosphere、hook
- 细纲（detailed_outline）：包含 sections 数组

### 3. 章节创建和更新

在 `incremental_insert_to_work` 方法中，确保：
- 大纲和细纲保存到 `chapter_metadata`：
  ```python
  chapter_metadata = {
      "outline": outline if outline else {},
      "detailed_outline": detailed_outline if detailed_outline else {},
  }
  ```

### 4. 卷支持

- 如果提供了 `volume_number`，确保章节的 `volume_number` 字段正确设置
- 卷的概念在数据库中通过 `Chapter.volume_number` 字段实现

### 5. 去重逻辑

- 检查章节是否已存在（按 `work_id` + `chapter_number` + `volume_number`）
- 如果存在，可以选择跳过或更新

## 使用示例

### 前端调用示例

```typescript
// 第一次分析（创建作品）
const result = await analyzeChapterByFile({
  file_name: "我的小说.txt",
  content: "第一章内容...",
  chapter_number: 1,
  volume_number: 1
});

// 第二次分析（继续插入）
const result2 = await analyzeChapterByFile({
  file_name: "我的小说.txt",  // 相同文件名，会找到已存在的作品
  content: "第二章内容...",
  chapter_number: 2,
  volume_number: 1
});
```

## 优势

1. **自动化**：无需手动创建作品，根据文件名自动处理
2. **连续性**：支持分多次分析同一文件的不同章节
3. **完整性**：每次分析都包含大纲和细纲
4. **灵活性**：支持卷的概念
5. **去重**：自动避免重复分析同一章节

## 注意事项

1. **文件名唯一性**：同一用户下，文件名应该唯一。如果同一文件有多个版本，可以考虑添加版本号或时间戳
2. **章节号管理**：确保章节号连续且不重复
3. **权限控制**：只能操作自己创建的作品
4. **错误处理**：如果AI返回的JSON格式不正确，应该给出清晰的错误提示
