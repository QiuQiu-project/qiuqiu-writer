# 多用户编辑文档同步机制说明

## 概述

系统采用 **WebSocket + HTTP API** 的混合架构实现多用户实时协作编辑，支持：
- 实时同步（WebSocket推送）
- 冲突检测与智能合并
- 版本控制
- 离线编辑支持

## 架构设计

### 1. 数据存储层

- **Redis**: 存储文档的当前状态和版本信息
  - Key: `doc:{document_id}`
  - Value: JSON格式的文档对象（包含content、version等）
  - TTL: 24小时（可配置）

- **PostgreSQL**: 存储同步历史记录（可选）
  - 表: `document_sync_history`
  - 用于审计和版本快照

### 2. 通信层

- **WebSocket** (`/ws`): 实时双向通信
  - 文档订阅/取消订阅
  - 实时更新推送
  - 心跳检测

- **HTTP API** (`/v1/sharedb/documents/sync`): 文档同步
  - 客户端主动推送更改
  - 冲突检测与合并
  - 版本号管理

## 工作流程

### 场景1: 用户A和用户B同时编辑同一文档

```
┌─────────────┐                    ┌─────────────┐
│   用户A     │                    │   用户B     │
└──────┬──────┘                    └──────┬──────┘
       │                                  │
       │ 1. 连接WebSocket                  │ 1. 连接WebSocket
       ├──────────────────────────────────┤
       │                                  │
       │ 2. 订阅文档 work_4_chapter_6     │ 2. 订阅文档 work_4_chapter_6
       ├──────────────────────────────────┤
       │                                  │
       │ 3. 编辑内容                      │
       │    "Hello" → "Hello World"      │
       │                                  │
       │ 4. 调用 sync API                 │
       │    POST /documents/sync          │
       │    {                             │
       │      doc_id: "work_4_chapter_6", │
       │      version: 1,                 │
       │      content: "Hello World"      │
       │    }                             │
       │                                  │
       │ 5. 服务器处理                    │
       │    - 更新Redis                   │
       │    - 版本号递增 (1→2)            │
       │    - 保存同步历史                │
       │                                  │
       │ 6. 广播更新                      │
       │    WebSocket推送                 │
       │    {                             │
       │      type: "document_synced",   │
       │      version: 2,                 │
       │      content: "Hello World"     │
       │    }                             │
       │                                  │
       │                                  │ 7. 收到WebSocket更新
       │                                  │    - 检查是否在编辑
       │                                  │    - 如果未编辑，立即应用
       │                                  │    - 如果正在编辑，标记待处理
       │                                  │
       │                                  │ 8. 编辑内容
       │                                  │    "Hello World" → "Hello World!"
       │                                  │
       │                                  │ 9. 调用 sync API
       │                                  │    POST /documents/sync
       │                                  │    {
       │                                  │      doc_id: "work_4_chapter_6",
       │                                  │      version: 2,  ← 使用最新版本
       │                                  │      content: "Hello World!",
       │                                  │      base_content: "Hello World"  ← 上次同步的内容
       │                                  │    }
       │                                  │
       │                                  │ 10. 服务器检测冲突
       │                                  │     - 服务器版本 = 2
       │                                  │     - 客户端版本 = 2
       │                                  │     - 但内容可能不同
       │                                  │     - 使用差异合并策略
       │                                  │
       │                                  │ 11. 合并结果
       │                                  │     - 版本号递增 (2→3)
       │                                  │     - 广播给所有用户
       │                                  │
       │ 12. 收到WebSocket更新            │
       │     - 应用合并后的内容           │
       │                                  │
```

### 场景2: 冲突处理

当两个用户同时修改同一段落时：

1. **用户A** 修改: `"Hello"` → `"Hello A"`
2. **用户B** 修改: `"Hello"` → `"Hello B"`
3. **服务器检测冲突**:
   - 服务器版本 > 客户端版本
   - 或内容不同但版本相同
4. **智能合并策略**:
   - **差异合并** (如果有base_content): 三路合并（base + server + client）
   - **智能合并** (如果没有base_content): 段落级别的合并
   - **最后写入获胜** (极端情况)

## 核心组件

### 1. ShareDBService (后端)

**文件**: `backend/src/memos/api/services/sharedb_service.py`

#### 关键方法

##### `join_collaboration(websocket, document_id, user_id)`
- 用户加入协作会话
- 将WebSocket连接添加到 `active_connections[document_id]`
- 发送当前文档状态给新用户
- 广播用户加入消息

##### `sync_document(document_id, version, content, base_content, user_id, db_session)`
- 同步文档内容
- 冲突检测与智能合并
- 版本号管理
- 保存同步历史（如果提供了db_session）
- **广播更新**给所有连接的客户端

##### `_broadcast_update(document_id, message)`
- 向所有订阅该文档的WebSocket客户端推送更新
- 自动清理断开的连接

#### 数据结构

```python
class ShareDBService:
    def __init__(self):
        self.active_connections: Dict[str, Set[WebSocket]] = {}  # 文档ID -> WebSocket连接集合
        self.document_locks: Dict[str, asyncio.Lock] = {}  # 文档锁，保证原子操作
        self.redis_client: Optional[redis.Redis] = None
```

### 2. ShareDBClient (前端)

**文件**: `frontend/src/utils/sharedbClient.ts`

#### 关键方法

##### `connect()`
- 建立WebSocket连接
- 自动重连机制
- 连接成功后自动订阅待订阅的文档

##### `subscribe(documentId, userId?)`
- 订阅文档更新
- 发送订阅消息到服务器
- 注册文档更新回调

##### `onDocumentUpdate(documentId, callback)`
- 注册文档更新回调
- 当收到WebSocket更新时触发

##### `syncDocumentState(documentId, content)`
- 同步文档状态到服务器
- 使用HTTP API (`/v1/sharedb/documents/sync`)
- 处理冲突和版本号

### 3. IntelligentSync Hook (前端)

**文件**: `frontend/src/utils/intelligentSync.ts`

#### 功能

- **防抖同步**: 用户停止输入后延迟同步
- **轮询更新**: WebSocket断开时的备用方案
- **智能应用更新**: 
  - 用户未编辑时立即应用
  - 用户正在编辑时延迟应用
- **版本去重**: 避免重复应用相同版本

## 冲突解决策略

### 1. 差异合并 (diff_based)

**条件**: 有 `base_content` 且不为空

**方法**: 三路合并
- Base: 上次同步的内容
- Server: 服务器当前内容
- Client: 客户端当前内容

**优势**: 最精确的合并，保留所有更改

### 2. 智能合并 (smart_merge)

**条件**: 没有 `base_content` 或为空

**方法**: 段落级别的智能合并
- 提取HTML段落
- 比较段落差异
- 合并不同段落

**优势**: 即使没有历史版本也能合并

### 3. 最后写入获胜 (last_write_wins)

**条件**: 无法合并时

**方法**: 使用较长的内容或服务器内容

**优势**: 保证数据一致性

## 版本控制

### 版本号规则

1. **新文档**: 版本号从 1 开始
2. **普通同步**: 版本号递增
3. **冲突合并**: 版本号 = max(server_version, client_version) + 1

### 版本号用途

- **冲突检测**: 比较客户端版本和服务器版本
- **去重**: 避免重复应用相同版本的更新
- **历史追踪**: 记录同步历史

## 性能优化

### 1. 防抖同步

前端使用防抖机制，用户停止输入后延迟同步：
- 默认延迟: 2秒
- 可配置

### 2. 批量更新

- 多个操作可以批量发送
- 减少网络请求

### 3. 本地缓存

- 使用 IndexedDB 缓存文档内容
- 离线时从缓存读取
- 在线时异步同步

### 4. 连接管理

- WebSocket连接复用
- 自动重连机制
- 心跳检测保持连接

## 测试方法

### 1. 基本同步测试

1. 打开两个浏览器窗口（或两个标签页）
2. 都访问同一个章节编辑页面
3. 在窗口A中输入内容
4. 观察窗口B是否自动更新（1-2秒内）

### 2. 冲突测试

1. 两个窗口同时编辑同一段落
2. 观察是否发生冲突
3. 检查合并结果是否合理

### 3. 离线测试

1. 断开网络
2. 编辑内容（应该保存到本地缓存）
3. 恢复网络
4. 检查是否自动同步

### 4. 多用户测试

1. 3个或更多用户同时编辑
2. 观察同步延迟
3. 检查最终内容一致性

## 常见问题

### Q1: 为什么两个用户看不到对方的更改？

**可能原因**:
1. WebSocket未连接
2. 未订阅文档
3. 网络问题

**解决方法**:
1. 检查浏览器控制台是否有连接错误
2. 确认已发送订阅消息
3. 检查网络连接

### Q2: 冲突合并结果不正确？

**可能原因**:
1. base_content不正确
2. 合并策略选择不当

**解决方法**:
1. 确保每次同步都传递正确的base_content
2. 检查合并策略日志

### Q3: 同步延迟过高？

**可能原因**:
1. 网络延迟
2. 服务器负载高
3. WebSocket连接不稳定

**解决方法**:
1. 检查网络状况
2. 查看服务器性能
3. 检查WebSocket连接状态
