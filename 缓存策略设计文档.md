# 缓存策略设计文档

## 一、策略选择：本地优先（离线优先）

### 1.1 为什么选择本地优先？

对于**写作应用**，推荐采用**本地优先（离线优先）**策略，原因：

#### ✅ 优势
1. **即时响应**：用户操作立即生效，无需等待网络
2. **离线可用**：完全离线状态下仍可使用
3. **性能提升**：减少网络请求，提升加载速度
4. **数据安全**：本地有备份，避免数据丢失
5. **用户体验**：流畅的编辑体验，无卡顿

#### ⚠️ 需要解决的问题
1. **数据一致性**：通过版本控制和后台同步解决
2. **冲突处理**：通过智能合并或用户选择解决
3. **数据同步**：通过后台同步机制解决

### 1.2 策略对比

| 策略 | 响应速度 | 离线支持 | 数据最新 | 网络依赖 | 适用场景 |
|------|---------|---------|---------|---------|---------|
| **本地优先** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | **写作应用** ✅ |
| 线上优先 | ⭐⭐ | ❌ | ⭐⭐⭐⭐⭐ | ⭐ | 实时数据应用 |

## 二、完整策略设计

### 2.1 读取策略（Read Strategy）

```
用户请求数据
  ↓
1. 检查本地缓存
  ├─ 有缓存 → 立即返回缓存数据 ✅
  │            ↓
  │         后台异步刷新（不阻塞）
  │            ↓
  │         更新缓存（如果服务器有更新）
  │
  └─ 无缓存 → 请求服务器
              ├─ 成功 → 缓存并返回
              └─ 失败 → 返回错误
```

**实现原则**：
- ✅ 优先使用本地缓存（快速响应）
- ✅ 后台异步刷新（不阻塞用户）
- ✅ 服务器失败时使用缓存（降级）

### 2.2 写入策略（Write Strategy）

```
用户保存数据
  ↓
1. 立即保存到本地缓存 ✅
  ↓
2. 防抖处理（300ms）
  ↓
3. 同步到服务器
  ├─ 成功 → 标记已同步
  └─ 失败 → 标记待同步，加入队列
  ↓
4. 后台重试（SyncManager）
```

**实现原则**：
- ✅ 本地优先保存（立即响应）
- ✅ 后台同步到服务器
- ✅ 失败时加入重试队列

### 2.3 冲突解决策略（Conflict Resolution）

```
检测到冲突（服务器版本 > 本地版本）
  ↓
1. 检查用户是否正在编辑
  ├─ 正在编辑 → 延迟应用，提示用户
  └─ 未编辑 → 自动应用服务器版本
  ↓
2. 应用更新
  ├─ 自动合并（如果可能）
  └─ 提示用户选择（如果冲突严重）
```

## 三、具体实现方案

### 3.1 章节列表 - 本地优先

```typescript
async listChapters(params): Promise<ChapterListResponse> {
  const cacheKey = `chapters_list_${params.work_id}_...`;
  
  // 1. 优先从本地缓存获取
  const cached = await localCacheManager.get(cacheKey);
  if (cached) {
    // 立即返回缓存数据
    // 后台异步刷新（不阻塞）
    this.refreshChaptersFromServer(params, cacheKey).catch(console.warn);
    return cached;
  }
  
  // 2. 缓存没有，从服务器获取
  return await this.listChaptersFromServer(params, cacheKey);
}
```

## 六、总结

**推荐策略：本地优先（离线优先）**

- ✅ 立即响应，提升用户体验
- ✅ 完全离线可用
- ✅ 减少网络请求，提升性能
- ✅ 数据安全，有本地备份

**配合机制**：
- 后台异步刷新确保数据最新
- 版本控制检测冲突
- 智能合并解决冲突
- 后台同步确保数据一致性
